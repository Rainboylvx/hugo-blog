# Haskell 99 多解文章修复设计

## 背景

P21-P99 已经补全正文，并在近期提交中增加了“方法一、方法二、方法三”等多解结构。评审发现其中一部分替代方法不能编译、没有满足原题语义、只是同一实现的语法改写，或者仍然是占位说明。Hugo 构建能够检查 Markdown，却不会检查代码块中的 Haskell，因此页面构建成功不能作为题解正确性的证明。

本次修复以 P01-P20 的文章组织方式为参考，保留 P21-P99 中已经正确且有教学价值的内容，逐题建立可验证的多解结构。

## 方案选择

### 方案一：只修复评审列出的错误

改动较少，但没有列入评审的文章仍可能只有一个有效实现，也无法保证整个系列达到统一标准。

### 方案二：逐题通过质量门禁修复（采用）

检查 P21-P99 的题意、签名、每个方法和测试。保留有效内容，删除或降级无效内容，为有效方法不足两种的题目补充真正不同的实现。改动范围可控，同时能形成一致的完成标准。

### 方案三：重写 P21-P99

统一性最高，但会覆盖已经写好的解释和作者手工补充内容，审查成本和回归风险都过大。

## 范围

### 包含

- 修复 P21-P99 中不能编译、结果错误、签名不一致或不满足题意的代码。
- 每篇文章至少保留两种正确且有实质差异的解法；没有合理第二种算法的题目，可以提供一个标准解法和一个不同抽象方式的完整实现，但必须说明二者关系。
- 恢复修订过程中误删的原有功能，例如 P61 的内部节点内容。
- 修复裸露代码、围栏错误、重复方法标题和缺失测试结果。
- 移除未声明的第三方依赖，优先使用 `base`、`containers` 和系列已有依赖。
- 修复或删除会再次生成错误内容的辅助脚本。

### 不包含

- 重写 P01-P20。
- 修改 Hugo 主题、栏目导航设计或全站样式。
- 为了凑数量而增加仅有变量名、排版或 `map`/列表推导差异的方法。
- 将大型题目的命令行交互、文件读写等非核心部分扩展成完整应用。

## 有效解法标准

一个章节只有同时满足以下条件，才可以命名为“方法”或“解法”：

1. 函数签名与文章采用的官方题意一致。
2. 对正常输入和文章声明的边界输入返回正确结果。
3. 代码完整，不含省略号、未定义名称或缺失的核心辅助函数。
4. 在文章声明的依赖环境中能够通过 GHC 类型检查。
5. 与其他方法存在可说明的差异，例如递归关系、遍历顺序、数据结构、搜索策略、复杂度或抽象工具不同。

近似算法、只覆盖部分输入的算法、伪代码和思路草图放入“延伸讨论”或“优化方向”，并明确限制，不能计入有效解法数量。使用不同语法表达同一算法时，合并到同一方法下面作为“等价写法”。

## 单篇文章结构

修复后的文章沿用当前栏目风格：

1. 题目描述和边界条件。
2. 官方函数签名及必要的数据类型。
3. 方法一：推荐实现，优先选择直接、易验证的版本。
4. 方法二：教学替代实现，明确与方法一的实质区别。
5. 可选方法三：只有在提供明显的性能、抽象或算法价值时增加。
6. 方法对比：说明思路、复杂度、依赖和适用场景。
7. 测试：所有有效方法共用相同的正常、边界和反例测试。

所有 Haskell 代码放在标有 `haskell` 的围栏代码块中。依赖、辅助类型和核心辅助函数在首次出现处给全，不能用“参考官方实现”代替代码。

## 验证策略

### Haskell 验证

- 将每个方法连同所需类型、导入和辅助函数放入独立临时模块，避免同名函数互相冲突。
- 使用 `ghc -fno-code` 做语法和类型检查。
- 使用 `runghc` 或 GHCi 表达式运行文章示例。
- 同一题的所有方法运行同一组测试；随机算法检查长度、元素来源和无重复等性质，而不是比较固定顺序。
- 搜索和图算法至少增加一个空输入、不可解输入或退化结构测试。

临时验证文件不提交到文章目录。若某题确需第三方包，必须在正文写明依赖并确认本机可用；否则改写为现有依赖可运行的版本。

### 内容验证

- 搜索 `...`、`TODO`、“参考官方实现”等占位文本。
- 检查所有裸露 Haskell 代码都进入围栏。
- 检查每篇有效方法数量，重复标题不计数。
- 检查测试中的预期结果与正文实现一致。
- 检查 P61 等文章没有因补多解而丢失原题功能。

### 站点验证

- 运行 `hugo`，确认 Markdown、front matter 和链接能够生成。
- 复查 Hugo 警告，区分既有配置警告和本次新增问题。
- 使用 `git diff --check` 检查行尾空白和补丁格式。

## 完成标准

- P21-P99 每篇至少有两个通过有效解法标准的方法。
- 不存在占位实现、未定义名称、错误签名或已知反例。
- 所有文章代码块格式正确，测试包含明确预期结果。
- 文章不依赖未声明且本机不可用的 Haskell 包。
- 阶段性 Haskell 类型检查与示例测试通过。
- Hugo 构建和 `git diff --check` 通过。
- 辅助脚本不会重新引入已修复的错误内容。

## 风险控制

- 图、树布局、数独、Nonogram 和填字等题目代码较长，按题独立验证，避免一次性机械生成。
- 当前文章可能采用与其他 Haskell 99 版本不同的签名；以文章已经链接的官方页面和当前系列共享类型为准，不混用旧版题号。
- 两种方法都正确但复杂度相同是允许的，关键是思考路径确实不同；无法形成合理第二种算法时，不制造虚假的“优化版”。
- 修复只触及 Haskell 99 文章、相关验证材料和生成脚本，不顺带整理其他博客内容。
