# `python-oj-learn` 全局技能设计

## 背景

`content/program_language/python/` 已经形成一组面向算法竞赛的 Python 学习文章，包含暴力验证、代码大模板、输入输出、排序、容器、生成器、`itertools`、随机数据和数学工具。用户希望在每次提交 OJ 题目时，AI 先检索这些文章，再给出可提交的 Pythonic 解法，并通过代码对照和模仿清单帮助用户快速掌握 Python。

技能还要发现知识库的真实缺口，但不能因为每道新题都自动扩写博客。遇到可迁移的 Python 知识缺口时，技能先完成题目，再提出新文章建议；只有用户明确确认后，才进入博客写作流程。

本技能安装为全局技能，不依赖当前工作目录。

## 方案选择

### 方案一：纯 `SKILL.md`

技能每次自行用 `rg` 搜索知识库。文件少，但不同调用可能采用不同搜索词和输出范围，检索行为不稳定。

### 方案二：技能加检索脚本（采用）

`SKILL.md` 负责题目分析、教学和缺口判断，标准库脚本负责列出文章、标题、标题层级和关键词片段。知识仍只保存在博客中，脚本不复制正文。

### 方案三：将文章快照复制进技能

技能可直接读取自己的参考文件，但博客更新后快照会过期，并产生双份知识源，因此不采用。

## 触发边界

### 应触发

- 用户发送 OJ、Codeforces、洛谷、AtCoder、LeetCode 等题目并要求 Python 解。
- 用户要求 Pythonic、用 Python 学 OJ、通过题目学习 Python、把 C++ 写法改成 Python。
- 用户发送完整竞赛题且没有指定语言，默认使用 Python 教学工作流。
- 用户同时要求思维分析和 Python 实现；可与专门的思维分析技能配合。

### 不应触发

- 用户明确只要 C++、Haskell、Rust 等其他语言。
- 用户只要求修复已有代码，并且代码语言不是 Python。
- 用户只问“题目本质、如何想到、为什么贪心正确”而没有要求 Python 代码；此时 `rainboy-brain` 更合适。
- 用户讨论普通业务 Python、Web 开发、数据分析，而不是 OJ 学习。

front matter 的 `description` 同时写明正向触发和这些邻近边界，避免只靠技能名称触发。

## 知识检索脚本

### 目标

提供稳定、低上下文成本的本地知识发现，不替代 AI 对相关文章的完整阅读。

### 接口

```bash
python scripts/search_python_notes.py --list
python scripts/search_python_notes.py "子集 combinations 生成器" --limit 8
python scripts/search_python_notes.py "heapq 优先队列" --json
python scripts/search_python_notes.py --root /custom/python/notes "BFS deque"
```

参数：

- 位置参数 `query`：一个或多个搜索词组成的查询字符串。
- `--list`：列出所有 Markdown 文章的标题、路径和二级标题。
- `--root`：覆盖知识库根目录；优先级高于环境变量。
- `--limit`：限制搜索结果，默认 8。
- `--json`：输出机器可读 JSON；默认输出紧凑文本。

### 行为

- 递归读取知识库中的 `.md` 文件。
- 解析 YAML front matter 中的 `title`，缺失时使用文件名。
- 提取二级和三级标题。
- 对标题、标题层级、正文关键词匹配进行简单加权评分。
- 返回文章绝对路径、相对路径、标题、分数、匹配标题和短片段。
- 使用 Python 标准库实现，无网络依赖。
- 查询为空且未指定 `--list` 时给出清晰用法错误。
- 根目录或 `_index.md` 不存在时返回非零退出码。

AI 不能只读搜索摘要就写解答。检索完成后，必须完整读取最相关的 2-4 篇文章；需要代码模板时再读取 `src/brute_force_template.py`。

## 每题工作流

### 1. 取得题目事实

优先读取：

1. 题目描述；
2. 输入输出格式；
3. 数据范围；
4. 样例；
5. 用户已有代码或本地题目文件。

题目信息不足以确定算法时先请求缺失内容，不猜测约束。

### 2. 检索本地 Python 知识

1. 读取 `_index.md`。
2. 根据算法形态和 Python 特性构造查询词。
3. 运行检索脚本。
4. 完整读取最相关的 2-4 篇文章。
5. 需要常用骨架时读取大模板源码。
6. 记录实际使用的文章，回答时列出。

### 3. 选择算法与 Python 表达

- 先根据约束选择复杂度满足要求的算法。
- Pythonic 不能以牺牲复杂度或正确性为代价。
- 优先采用知识库已经解释的清晰写法。
- 不为了“一行代码”隐藏关键状态变化。
- 考虑 Python 的递归深度、整数特性、输入速度、对象开销和标准库语义。

### 4. 生成代码

- 必须给一份完整、可提交的 Pythonic 正解。
- 代码包含所需导入、输入、输出和边界处理。
- 暴力版本足够短且有教学价值时，再提供 `brute()`；不强行凑第二份代码。
- 变量名和函数边界应帮助读者模仿，不使用无意义缩写堆砌技巧。

### 5. 验证

- 有样例时实际运行全部样例。
- 能快速写小规模暴力时，使用固定种子随机数据比较正解与暴力。
- 检查空输入、最小输入、重复值、极值等相关边界。
- 不为了验证而修改用户题目仓库。
- 无法运行时明确写“未验证”、原因和剩余风险。

## 教学输出

默认按以下结构输出，简单题可合并短小章节，但不能省略代码、复杂度和验证状态：

1. **题意与约束**：指出决定算法的条件。
2. **算法思路**：核心观察、正确性要点和复杂度。
3. **Pythonic 正解**：完整可提交代码。
4. **Python 知识映射**：本题用到的语法、标准库和实际读取的本地文章。
5. **C++ → Python 对照**：只比较本题出现的写法。
6. **模仿清单**：列出 2-4 个值得反复手写的片段。
7. **可选暴力版本**：仅在短小且有教学价值时出现。
8. **验证结果**：列出样例、随机校验或未验证原因。
9. **知识缺口建议**：只有确实存在时出现。

本地文章用绝对文件链接引用，并说明该文章具体支持了哪种写法。不要把文章列表机械地全部贴出。

## Python 知识缺口

### 判定标准

只有缺少可迁移的 Python 知识时才建议新增文章，例如：

- 新的标准库或容器用法；
- 多题可复用的 Pythonic 表达模式；
- Python 特有的输入输出、递归限制、对象开销、精度或调试问题；
- 多道 OJ 题都能复用的 Python 实现模板。

新的算法原理本身不是 Python 知识缺口。例如遇到线段树时，应先区分缺的是线段树算法知识，还是“Python 如何高效实现线段树”的通用经验。

### 建议格式

先完成当前题目，然后给出：

- 拟新增文章标题；
- 为什么现有文章不足；
- 可迁移到哪些题型；
- 文章提纲；
- 建议文件名和目标目录。

最后询问用户是否确认创建。未经明确确认，不创建或修改博客文章。

### 用户确认后

- 使用可用的 `rainboy-hugo-blog-writing` 技能和环境要求的创作设计流程。
- 写入 `/home/rainboy/mycode/hugo-blog/content/program_language/python/`。
- 文章讲通用 Python 模式，当前题目只作为一个例子。
- 默认 `draft: true`。
- 更新 Python `_index.md`。
- 执行文章 Python 代码、相对链接和 Hugo 构建验证。
- 回到当前题目，说明新文章如何支持解法。

## 与其他技能协作

- `rainboy-brain`：负责题目本质、发现路径和关键证明；本技能负责 Python 代码教学。
- `rbook-http`：用户要求根据 rbook 文章或模板解题时，由它提供算法知识；本技能仍负责 Python 知识映射。
- `rainboy-hugo-blog-writing`：用户确认新增 Python 文章后负责博客写作规范。

不要重复其他技能已经完成的分析；在最终回答中合并成一条连贯的学习路径。

## 评测用例

### 用例一：现有知识足够

题目：给定 $n\le18$ 个整数和目标值 $S$，判断是否存在一个子集之和等于 $S$。要求 Pythonic 解答和适合模仿的解释。

预期：

- 使用位掩码、`combinations` 或 `product` 中合理的一种。
- 引用暴力验证、生成器、`itertools` 或模板中的实际相关文章。
- 给出 $O(2^n\cdot n)$ 或相应实现的准确复杂度。
- 运行样例。
- 不建议新增 Python 文章，因为知识库已经覆盖。

### 用例二：存在 Python 知识缺口

题目：给定 $n,m\le2\times10^5$ 的带正权有向图，求 1 到 $n$ 的最短距离，不可达输出 `-1`。要求 Pythonic 解答和适合模仿的解释。

预期：

- 使用邻接表和 `heapq` Dijkstra，复杂度 $O((n+m)\log n)$。
- 引用输入输出、容器和大模板中实际相关的内容。
- 运行样例，并在可行时做小图校验。
- 先完成解答，再建议一篇通用的 Python `heapq`/优先队列文章。
- 不自动创建博客文章。

## 评测方法

将两条测试分别运行：

- `with_skill`：明确加载 `/home/rainboy/.agents/skills/python-oj-learn/SKILL.md`；
- `without_skill`：相同题目，不加载技能。

客观断言：

- 正解代码能通过给定样例；
- 算法和复杂度满足约束；
- 本地引用指向存在的文件；
- 输出包含 Python 知识映射、C++ 对照、模仿清单和验证状态；
- 用例一不提出无谓的新文章；
- 用例二提出建议但没有自动写博客。

结果保存到 `/home/rainboy/.agents/skills/python-oj-learn-workspace/iteration-1/`。有技能和无技能的运行同时启动，保存 token 和耗时，完成后评分并聚合 `benchmark.json`。使用 `skill-creator/eval-viewer/generate_review.py --static` 生成静态 HTML，供用户审阅实际输出和量化结果。
